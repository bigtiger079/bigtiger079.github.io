笔记

JNI - Java Native Interface;

如何载入*.so

VM的角色

由于Android的应用层级类别都是以Java撰写的, 这些Java类转译为Dex形式的bytecode之后,必须依赖Dalvik虚拟机来执行它, VM在Android平台里扮演很重要的角色

此外,在执行Java类的过程中,如果Java类需要与JNI本地模块沟通,VM就会去加载本地模块,然后让Java的函数顺利调用到本地模块的函数.此时,VM扮演着桥梁的角色, 让Java与本地模块能透过标准的JNI接口而相互沟通

Java层的类是在VM上执行的, 而本地模块则不是在VM上执行的, 那么Java程序如何要求VM去加载所指定的C模块呢?
可以使用下述指令:

System.loadLibrary(*.so的文件名);

例如, NativeJniAdder类:

public class NativeJniAdder{
    static{
        try{
            Log.i("JNI", "Trying to load libNativeJniAdder.so");
            System.loadLibrary("NativeJniAdder");
        }catch(UnsatisfiedLinkError e){
            Log.i("JNI", "cloud not load libNativeJniAdder.so");
        }
    }

    public static native long newObject();
    public static native long execute(long refer, int digit_1, int digit_2);
}

jni如下:

#include "Adder.h"
#include "com_xxx_xxx_NativeJniAdder.h"

JNIEXPORT jlong JNICALL Java_com_xxx_xxx_NativeJniAdder_newObject(JNIEnv *env, jclass c){
    Adder* ar = (Adder*)AdderNew();//创建一个C对象
    return (jlong)ar;
}

JNIEXPORT jlong JNICALL Java_com_xxx_xxx_NativeJniAdder_execute(JNIEnv *env, jclass c, jlong refer, jint digit_1, jint digit_2){
    Adder* pa = (Adder*)refer;//转成对象的指针
    long result = pa->exec(digit_1, digit_2);
    return result;
}

定义Adder类(Adder.h)

typedef struct Adder Adder;
struct Adder{
    int (*exec)(int a, int b);
};

struct Adder *AdderNew(){
    struct Adder *t = (Adder*)malloc(sizeof(***));
    t->exec = my_exec;
    return (void *)t;
}
static int my_exec(int a, int b){
    return (a+b);
}

这个JNI接口定义类别含有2个函数: newObject()和execute()
其中, newObject()函数诞一个Adder对象,并且将该对象的指针传递回来给Java程序
而exectue()函数的refer参数,是用来让Java程序能将对象指针传进去给execute()函数,此时execute()就能借由该指针而调用到先前的newObject()函数所诞生的那个对象

VM调用<Tn>本地函数时, 将Java层对象指针(pointer)传给<Tn>
配上<Tn>后,<Tn>可以将C/C++对象指针会传到Java层

由于这些Java和C代码都在同一进程里执行,所以指针都是可以互传

*.so的入口函数: JNI_OnLoad()

Java代码在VM上执行
在执行Java代码过程中,如果需要与本地代码(*.so)沟通, VM就会把*.so视为插件<Tn>而加载到VM里
然后让Java函数调用到这插件里的C函数

插件是由VM来管理的,实际上VM是*.so插件的管理器(Plug-in Manager)
Java与C函数的调用,也是通过VM来对接的

在执行System.loadLibrary()函数时,VM会反向调用*.so里的JNI_Onlaod()函数.用途有二:

  1. VM询问此*.so使用JNI版本编号
  2. VM要求*.so做一些初期设定工作(Initialization),例如登记<函数名称表>

在Android的/system/lib/libmedia_jni.so里,就提供了JNI_Onload()函数,其程序代码为:

//define LOG_NDEBUG 0
#definde LOG_TAG "MediaPlayer_JNI"
//......
jint JNI_Oload(JavaVM* vm, void* reserved){
    JNIEnv* env =NULL;
    jint result = -1;
    if(vm -> GetEnv((void**)&env, JNI_VERSION_1_4) != JNI_OK){
        LOGE("ERROR:GetEnv failed\n");    goto bail;     
     }
    assert(env != null);
    if(register_android_media_MediaPlayer(env)<0){
        LOGE("ERROR:MediaPlayer native registration failed\n");    
        goto bail;     
    }
    /* success -- return valid version number */
    result = JNI_VERSION_1_4;

bail: return result;
}
//KTHXBYE

此函数回传JNI_VERSION_1_4值给VM, 于是VM知道了其所使用的JNI版本了
此外, JNI_OnLoad()函数也做了一些初期的动作,例如指令:

if(register_android_media_MediaPlayer(env)<0){
    LOGE("ERROR:MediaPlayer native registration failed\n");    
    goto bail;     
}

就将此*.so的<函数名称表>登记到VM里,以便能加快后续调用本地函数的效率

当VM释放该C模块时, 则会调用JNI_OnUnload()函数来进行善后清除动作

registerNativeMethods()函数的用途

Java类通过VM调用到本地函数
一半是依赖VM去寻找.so里的本地函数.如果需要连续调用很多次, 每次都需要寻找一遍,会花很多时间
此时,将此.so的<函数名称表>登记到VM里, 例如在Android的/system/lib/libmedia_jni.so里的程序代码片段如下:

//define LOG_NDEBUG 0
#definde LOG_TAG "MediaPlayer_JNI"
//......
static JNINativeMethod getMethods[]={
{"setDataSource","(Ljava/lang/String;)V",(void*)android_media_MediaPlayer_setDataSource},
{"setDataSource","(Ljava/io/FileDescriptor;JJ)V",(void*)android_media_MediaPlayer_setDataSourceFD},
{"prepare","()V",(void*)android_media_MediaPlayer_prepare},
{"prepareAsync","()V",(void*)android_media_MediaPlayer_prepareAsync},
{"_start","()V",(void*)android_media_MediaPlayer_start},
{"_stop","()V",(void*)android_media_MediaPlayer_stop},
......
};

第一个表示函数的别名,在Java里面使用
中间代表函数的参数和回传值
第三个表示函数的本名

//......
static int register_android_media_MediaPlayer(JNIEnv *env){
    ......
    return AndroidRuntime::registerNativeMethods(env, "android/media/MediaPlayer", getMethods, NELEM(getMethods));
}
//......
jint JNI_OnLoad(JavaVm* vm, void* reserved){
    ......
    if(register_android_media_MediaPlayer(env)<0){
        LOGE("ERROR: MediaPlayer native registration failed\n");
        goto bail;
    }
    //......
}

JNI_OnLoad()调用 register_android_media_MediaPlayer()函数,此时就调用到
AndroidRuntime::registerNativeMethods()函数,向VM登记gMethods[]表格

登记gMethods[]表格的用途有二:

  1. 更有效率的去找到C函数
  2. 可在执行期间弹性进行抽换

由于gMethods[]是一个<名称, 函数指针>对照表,在程序执行是,可多次调用registerNativeMedhods()来更换本地函数的指针,而达到弹性抽换本地函数的目的.
